## 1. Тезис 2026: что изменилось

**Коротко:** тяжёлые 20-страничные спеки для AI-фич мертвы. Требования стали короткими, итеративными и изначально промпт-нативными — пишутся как промпт, а не как документ. Качество требований — главный рычаг окупаемости GenAI.

Три сдвига, которые всё объясняют:

1. **Спек = промпт.** Product School продвигает метод **PRO — Prototyping Requirements One-Pager**: ≤500 слов, 8 секций, изначально пишется как промпт для LLM. Microsoft AI group признал 20-страничные спеки устаревшими для AI-фич *(⚠️ требует перепроверки первоисточником)*. Метафора Aparna Sinha (Vercel): «building in AI is like building in an earthquake» — спек должен быть коротким и пересматриваться каждый цикл *(⚠️ цитата требует перепроверки)*.

2. **ROI упирается в требования, а не в модели.** MIT NANDA: ~95% организаций получают нулевой ROI от GenAI из-за бездумного внедрения *(⚠️ цифра требует перепроверки)*. Вывод: качество требований — это и есть рычаг окупаемости. Не «какую модель купить», а «какую проблему решаем и какие у неё edge-cases».

3. **Velocity = moat.** Elena Verna (Lovable): velocity как конкурентный рывок, полная автономия с AI-инструментами *(⚠️ требует перепроверки)*. Linear Method кодифицирует это: короткие project specs, cycles, named owners, «diff = progress», AI-фичи встроены в тулзы (Asks/Intake/Insights).

**Роль года — AI Builder.** Product School сертификации 2026: Vibe Coding, Claude Code, AI Evals, Advanced AI Agents *(⚠️ список сертификаций требует перепроверки)*.

**Главный практический вывод:** спецификация AI-фичи должна быть достаточно короткой, чтобы её можно было переписать за цикл, и достаточно структурированной, чтобы стать прямым входом в промпт/прототип. Отсюда — формат PRO.

## 2. Decision framework — как выбрать

Лучший справочник бесполезен, если не отвечает «что мне взять прямо сейчас». Вот дерево выбора.

### 2.1 Какую методику взять под задачу

```
Что у тебя на входе?
│
├─ «Есть размытая проблема, непонятно кому и зачем»
│    → JTBD / ODI  +  Continuous Discovery (Opportunity Solution Tree)
│
├─ «Есть смелая идея, надо быстро проверить и продать внутри»
│    → Working Backwards / PR-FAQ  (Amazon)
│
├─ «Команда маленькая, нужен фиксированный кусок работы на 6 недель»
│    → Shape Up (Pitch + Bet + Cycle)
│
├─ «Сложная предметка, много интеграций, нужен контракт»
│    → DDD + Event Storming  +  RFC/Design Doc  +  OpenAPI
│
├─ «Нужно превратить требования в проверяемое поведение»
│    → BDD / Example Mapping / Gherkin
│
└─ «Надо приоритизировать бэклог»
     → RICE / WSJF  (+ Kano для классификации ожиданий)
```

### 2.2 Какой формат документа взять

| Ситуация | Формат | Объём |
|---|---|---|
| AI-фича, быстрый цикл, итеративно | **PRO one-pager** | ≤500 слов |
| Классическая фича, нужна выжимка | **Lean PRD** | 1–2 страницы |
| Продать愿景 наверх/проверить спрос | **PR-FAQ** (Amazon) | 1 страница + FAQ |
| 6-недельный бет Shape Up | **Pitch** | 1–2 страницы |
| Сложная техническая инициатива | **RFC / Design Doc** | 3–10 страниц |
| Архитектурное решение, оставить след | **ADR** | 0.5–1 страница |
| Карта возможностей из исследований | **Opportunity Solution Tree** | 1 диаграмма |
| Нарезка релиза по шагам пользователя | **Story Map** | 1 диаграмма |

### 2.3 Когда какую AI-поддержку подключать

| Этап | Что усиливает AI | Осторожно |
|---|---|---|
| Discovery / синтез интервью | C1 (OST-синтез), research-synthesis тулзы | галлюцинации; всегда сверяй с сырыми заметками |
| Написание спека | C2 (генерация PRO), C7 (self-review) | AI сглаживает острые углы — возвращай конкретику |
| Проектирование AI-фичи | C3 (симуляция флоу), C8 (UI-промпт) | симуляция ≠ прод; edge-cases надо прогонять руками |
| Валидация | C4 (pre-mortem), C5 (Gherkin) | не замена пользовательскому тестированию |
| Синхронизация артефактов | C6 (дрифт-детекция) | проверяй семантику, не только строки |

## 3. Методологии (карточки)

Каждая карточка: **когда · вход · выход · AI-рычаг · когда НЕ брать**.

### 3.1 JTBD / ODI (Jobs-to-be-Done / Outcome-Driven Innovation)
- **Когда:** продукт теряет фокус; нужно понять, зачем люди «нанимают» продукт.
- **Вход:** интервью, переключения между решениями (switch interviews).
- **Выход:** Job Statement + Forces of Progress + Desired Outcomes (metrics).
- **AI-рычаг:** C1 синтезирует интервью в гипотезы jobs; **не** заменяет интервью.
- **Не брать:** когда рынок и потребности уже стабильны и понятны — избыточно.

### 3.2 Continuous Discovery + Opportunity Solution Tree (Teresa Torres)
- **Когда:** нужна непрерывная проверка гипотез, а не разовый бэклог.
- **Вход:** еженедельные интервью → opportunities → solutions → experiments.
- **Выход:** дерево Возможностей (из проблемы → ветки возможностей → решения → эксперименты).
- **AI-рычаг:** C1 строит OST из сырых заметок интервью.
- **Не брать:** разовый MVP с фиксированным скоупом — OST избыточен.

### 3.3 Working Backwards / PR-FAQ (Amazon)
- **Когда:** есть смелая идея; надо сформулировать «какой будет press release».
- **Вход:** видение результата.
- **Выход:** Press Release (от лица будущего клиента) + FAQ (внутренний + внешний).
- **AI-рычаг:** AI помогает написать «вкусный» PR и предсказать вопросы FAQ.
- **Не брать:** мелкие инкрементальные правки — слишком тяжёлая церемония.

### 3.4 Shape Up (Basecamp)
- **Когда:** маленькая команда, фиксированные 6-недельные циклы (или 2-недельные).
- **Вход:** проблема достаточно зрелая для «bet».
- **Выход:** Pitch (проблема + решение на уровне аппетита + «no-baggage» scope) → Bet → Cycle → Hill chart.
- **AI-рычаг:** AI помогает сузить scope («что выкинуть из Pitch»).
- **Не брать:** продуктовый континуум с непрерывным деплоем — циклы будут мешать.

### 3.5 Impact Mapping
- **Когда:** связать бизнес-цель → актёры → воздействия → фичи.
- **Вход:** бизнес-цель (например, «−20% обращений в саппорт»).
- **Выход:** дерево цели → персоны → воздействий → deliverables.
- **AI-рычаг:** AI генерирует варианты воздействий; человек отбирает.
- **Не брать:** когда цели ещё нет — строить карту по выдуманной цели бессмысленно.

### 3.6 User Story Mapping (Jeff Patton)
- **Когда:** нужен релизный план по шагам пользователя, с версиями (v1/v2/v3).
- **Вход:** user activities → steps → tasks.
- **Выход:** карта с «backbone» сверху и приоритезацией сверху-вниз.
- **AI-рычаг:** AI раскладывает user journey на steps.
- **Не брать:** backend-only / infra без пользовательского пути.

### 3.7 DDD + Event Storming
- **Когда:** сложная предметка, много bounded contexts, интеграции.
- **Вход:** domain experts + sticky-note сессия (events, commands, aggregates).
- **Выход:** карта домена, bounded contexts, агрегаты.
- **AI-рычаг:** AI помогает назвать агрегаты/события; **не** заменяет экспертов.
- **Не брать:** CRUD-приложение без сложной логики — оверкилл.

### 3.8 BDD / Example Mapping / Gherkin
- **Когда:** нужно превратить требования в проверяемое поведение, мостик к автотестам.
- **Вход:** feature + правила + примеры.
- **Выход:** `.feature` файлы (Given/When/Then).
- **AI-рычаг:** C5 генерирует Gherkin из feature-описания.
- **Не брать:** исследовательская фича, где правила ещё не устоялись.

### 3.9 Dual-Track Agile
- **Когда:** нужно одновременно делать discovery и delivery, не сливая их.
- **Вход:** две параллельных дорожки с синхронизацией.
- **Выход:** непрерывный поток гипотез на одной дорожке и фич на другой.
- **AI-рычаг:** AI ускоряет обе дорожки; опасность — ускорить delivery быстрее discovery.
- **Не брать:** solo/маленький проект — избыточно.

### 3.10 Lean Startup
- **Когда:** высокая неопределённость, надо «строить-измерять-учиться».
- **Вход:** гипотеза + минимальный эксперимент.
- **Выход:** validated learning, pivot-or-persevere решение.
- **AI-рычаг:** AI помогает спроектировать MVP и метрику.
- **Не брать:** зрелый продукт на optimization-фазе.

### 3.11 Приоритизация: CIRCLES / Kano / RICE / WSJF
- **CIRCLES** — каркас для ответа «что строить» (Criteria, Insights, Reach, …).
- **Kano** — классификация ожиданий (basic/performance/delight).
- **RICE** — Reach × Impact × Confidence ÷ Effort.
- **WSJF** — Cost of Delay ÷ Job Size (для больших бэклогов/SAFe).
- **AI-рычаг:** AI помогает собрать оценки и подсветить заниженный Confidence.

## 4. Форматы документов + копируемые шаблоны

### 4.1 PRO one-pager (Product School) — шаблон

> 8 секций, ≤500 слов. **Пишется изначально как промпт.** Это самый ценный формат для AI-фич.

```text
# PRO: {название фичи}

## 1. Problem & goal
- Проблема: {кого, как часто, как больно, в деньгах/времени}
- Цель: {измеримый исход, напр. «−30% обращений в саппорт по теме X за квартал»}

## 2. Users & golden path
- Пользователи: {персоны/роли}
- Golden path (3–5 шагов): {идеальный сценарий «happy path»}

## 3. Key requirements
- {R1} … {R2} … {R3}  (конкретные, проверяемые)

## 4. AI considerations
- Tone/voice: {как отвечает}
- Latency: {порог, напр. P95 < 2s до первого токена}
- Accuracy/grounding: {чем обусловливаем ответ; RAG/источник}
- Safety: {что запрещено; guardrails}

## 5. Assumptions & constraints
- Предположения: {…}
- Ограничения: {бюджет, данные, регуляторика, приватность}

## 6. Edge cases & risks
- {EC1} → реакция: {…}
- {EC2} → реакция: {…}
- Риски + митигация: {…}

## 7. Success metrics
- Первичная: {one metric that matters}
- Вторичные: {2–3}
- Anti-metric (что НЕ должно вырасти): {напр. escalation rate}

## 8. Next steps
- {что делаем на этой неделе} → {кто owner} → {какой артефакт}
```
**Формат вывода:** один markdown-файл ≤500 слов. Если не укладываешься — сокращай раздел 3, а не 4 и 6.

### 4.2 Lean PRD — шаблон

```text
# Lean PRD: {фича}
**Context:** 1–2 строки, зачем.
**Goals / Non-goals:** что входит и что НЕ входит.
**User stories:** As {persona}, I want {…}, so that {…}.
**UX & flows:** ссылка на прототип/Figma + ключевые экраны.
**Requirements (must):** {минимальный набор}.
**Metrics:** {первичная + anti-metric}.
**Open questions:** {что ещё неизвестно}.
**Out of scope:** {явно}.
```

### 4.3 PR-FAQ (Amazon, Working Backwards) — шаблон

```text
# Press Release (от лица будущего, 1 страница)
- Заголовок: {что клиенты смогут делать}.
- Subhead: {кто целевой клиент, в чём выгода}.
- Лид: {проблема} → {решение}.
- Quote от «клиента» / руководителя.
- Как это работает (3–5 пунктов).
- Цитата пользователя (вымышленная, но реалистичная).
- Призыв/доступность.

# FAQ
## Внешний (клиент): {5–7 вопросов, которые реально зададут}
## Внутренний (команда): {технические, риски, метрики, costs}
```

### 4.4 Pitch (Shape Up) — шаблон

```text
# Pitch: {название}
**Problem:** кого и как болит (конкретика, не абстракции).
**Appetite:** small (2 нед) / big (6 нед).
**Solution:** набросок решения (Fat Marker sketches / текст), что делаем.
**Rabbit holes:** где можно застрять — заранее пометить.
**No-gos:** что НЕ делаем (scope-гигиена).
```

### 4.5 RFC / Design Doc — каркас
Context · Goals/Non-goals · Background · Proposed design · Alternatives considered · API/Data model · Security & privacy · Rollout/rollback · Open questions · References.

### 4.6 ADR (Architecture Decision Record) — каркас
```text
# ADR-{NN}: {решение}
**Status:** proposed|accepted|superseded by [[ADR-XX]]
**Context:** {почему решаем}
**Decision:** {что решили}
**Consequences:** +/−/нейтрально
```

### 4.7 Opportunity Solution Tree — каркас
Верх: **желаемый бизнес-результат** → ветки 2: **возможности** (opportunities) → ветки 3: **решения** → ветки 4: **эксперименты**. Одна ветка решений проверяется одним экспериментом с одной метрикой.

### 4.8 Story Map — каркас
Горизонталь (backbone): **user activities** слева направо. Вниз от каждого — **steps → tasks**. Релизные срезы горизонтальными линиями (v1/MVP сверху, v2/v3 ниже).

## 5. AI-инструменты: ландшафт по категориям

> Категории устойчивы. **Названия продуктов и их возможности быстро меняются — проверяйте актуальность перед внедрением.** Указан реальный use-case, а не маркетинг.

### 5.1 LLM-ассистенты (написание, рефакторинг, спарринг)
- **Claude / Claude Code, ChatGPT (agent mode), Gemini, GitHub Copilot.**
- Use-case: генерация PRO/PRD из заметок, self-review, рефакторинг спека, мок-интервью с «персоной».
- Осторожно: галлюцинации цитат/метрик; всегда сверять с источником.

### 5.2 Text → UI / код (прототип за часы)
- **v0 (Vercel), Lovable, Bolt.new, Framer AI, Visily, Uizard.**
- Use-case: превратить PRO в кликабельный прототип в тот же день; C8 даёт под это промпт.
- Осторожно: прототип ≠ прод; a11y и дизайн-система часто нарушаются — прогоняй `ux-design-review`.

### 5.3 Research synthesis (интервью → инсайты)
- **Dovetail AI, Marvin, Dscout, Notably.**
- Use-case: кластеризация интервью, теги, поиск по «кто ещё говорил про X», построение OST.
- Осторожно: AI-теги фиксируют предвзятость исследователя; оставляй ссылку на исходную цитату.

### 5.4 Спек-хабы с AI (где живёт спек + AI поверх)
- **Linear (Method), Notion AI, Confluence + Rovo, Productboard AI, ClickUp Brain.**
- Use-case: спек живёт рядом с тикетами; AI генерирует задачи из спека, ищет дубли, суммаризует комментарии.
- Осторожно: Atlassian Rovo/Linear AI дают агенту по базе — следи за **excessive agency** и правами.

### 5.5 Контракты (single source of truth)
- **OpenAPI / GitHub Spec Kit, drf-spectacular, Speakeasy.**
- Use-case: backend-контракт — источник правды; frontend-типы генерируются из него (см. AGENTS.md «Cross-Stack Contract Rules»).
- Осторожно: ручные дубли контракта — главный источник дрифта; C6 его ловит.

### 5.6 Eval / качество промптов и фич
- **LangSmith, Braintrust, Promptfoo.**
- Use-case: тестовые наборы для AI-фичи, регрессии промпта, оффлайн-оценки до прода.
- Осторожно: нет эвалов — нет возможности отличить улучшение от регресса; для AI-фичи это обязательно.

## 6. Промпт-библиотека C1–C8

## 7. Воркфлоу (5 схем)

### WF-1. Discovery → PRO (день)
1. 2–5 интервью / анализ метрик → сырые заметки.
2. **C1** → дерево возможностей.
3. Выбираешь одну ветку (opportunity + solution).
4. **C2** → PRO one-pager.
5. **C7** → self-review, правишь.
6. Артефакт: утверждённый PRO.

### WF-2. PRO → прототип (часы)
1. **C8** → промпт для UI-генератора.
2. v0/Lovable → кликабельный прототип.
3. **C3** → симуляция AI-флоу (если фича диалоговая).
4. Артефакт: прототип + лог симуляции.

### WF-3. AI-фича: симуляция до кода
1. PRO готов.
2. **C3** прогоняешь 5–10 сценариев, включая adversarial.
3. **C4** → pre-mortem + OWASP-маппинг.
4. Закрываешь найденные дыры в спеке (latency budget, guardrails, fallback).
5. **C5** → Gherkin на ключевые поведения.
6. Эвалы (LangSmith/Braintrust/Promptfoo) — test set до разработки.
7. Артефакт: спек с guardrails + эвал-набор.

### WF-4. Синхронизация артефактов (регулярно)
1. **C6** → дрифт-отчёт PRD↔дизайн↔тикеты.
2. Закрываешь расхождения.
3. Перегенерируешь frontend-типы из OpenAPI (контракт = источник правды).
4. Частота: раз в цикл/спринт.

### WF-5. Continuous Discovery (постоянно)
1. Еженедельные интервью.
2. **C1** обновляет OST.
3. Решения проверяются экспериментами с одной метрикой.
4. Anti-pattern: discovery и delivery слиты в одну дорожку → разделяй (Dual-Track).

## 8. Разобранный пример: Discovery → PRO

Сквозной кейс на выдуманном продукте, чтобы увидеть артефакты каждого шага.

**Продукт:** AI-ассистент триажа тикетов поддержки (классифицирует входящий тикет, предлагает ответ, эскалирует спорные).

### Шаг 1. Сырые заметки (фрагмент)
```text
[Интервью 1, head of support, 30 мин]
- 60% тикетов — однотипные («как сбросить пароль», «где счёт»).
- Агенты устают к концу дня, время первого ответа растёт.
- Боятся автоответов: «если бот ответит невпопад — клиент злится больше».
- Метрика, за которую платят: FRT (first response time), CSAT.
[Метрика]
- P95 FRT сейчас 4ч 10м. Цель бизнеса: < 1ч.
```

### Шаг 2. C1 → дерево возможностей (фрагмент вывода)
```text
Бизнес-результат: снизить P95 FRT с 4ч10м до <1ч за квартал.

Возможности:
  O1. Однотипные тикеты можно отвечать без человека. [цитата И1; метрика 60%]
  O2. Агенты устают → ухудшение к вечеру. [цитата И1]
  O3. Риск неверного автоответа злит клиента. [цитата И1]
Решения для O1:
  S1. AI-черновик ответа агенту на approve. → Эксперимент: доля тикетов с ответом <5м, метрика adoption%.
  S2. Полный автоответ для top-5 тем.         → Эксперимент: CSAT на автоответах vs ручных.
⚠️ Нет уликов: {доля, где auto-answer безопасен по регуляторике} — ТРЕБУЕТСЯ проверка.
Противоречия: {хотят скорость} vs {боятся неверного ответа} → tension в дизайне.
```

### Шаг 3. Выбираем ветку: O1 → S1 (AI-черновик на approve). Метрика: adoption% + FRT.

### Шаг 4. C2 → готовый PRO (результат)
```text
# PRO: AI-черновик ответа для агентов поддержки

## 1. Problem & goal
- Проблема: 60% тикетов однотипны; P95 FRT = 4ч10м; агенты устают к вечеру.
- Цель: P95 FRT < 1ч за квартал; ≥40% тикетов получают AI-черновик.

## 2. Users & golden path
- Пользователи: агент поддержки (внутренний).
- Golden path: тикет открыт → система предложила черновик → агент одобрил/правит → отправлено (FRT фиксируется) → CSAT-запрос.

## 3. Key requirements
- R1. Черновик появляется <5с после открытия тикета.
- R2. Источник ответа — только база знаний (RAG), с цитированием статьи.
- R3. Агент всегда может править и обязан подтвердить отправку.

## 4. AI considerations
- Tone: деловой, краткий, без эмодзи.
- Latency: P95 <5с до показа черновика.
- Accuracy: grounding только на KB; если confident < порога — не предлагать, отдать агенту пустой.
- Safety: не передавать PII в логи; no off-topic (только тема тикета).

## 5. Assumptions & constraints
- Предполагаем: KB актуальна и чистая.
- Ограничения: GDPR; данные не уходят во внешний LLM без zero-retention.

## 6. Edge cases & risks
- EC1: KB молчит → не предлагать черновик, не галлюцинировать.
- EC2: тикет злой/угроза → эскалация человеку, флаг.
- EC3: confident высокий, но ответ неверный → ручной approve страховка; метрика wrong-answer-rate.

## 7. Success metrics
- Первичная: P95 FRT < 1ч.
- Вторичные: adoption% ≥40; среднее время правки черновика.
- Anti-metric: CSAT не должен упасть; escalation rate не растёт.

## 8. Next steps
- На неделе: определить top-5 тем, собрать test set 50 тикетов, запустить эвал. Owner: {имя}.
```

### Шаг 5. C7 → self-review (фрагмент вердикта)
```text
- Проблема операциональна: 2/2 (есть % и FRT).
- Golden path конкретный: 2/2.
- Требования проверяемы: 2/2.
- AI-секции заполнены: 2/2.
- Edge cases: 2/2 (≥3).
- Метрики + anti-metric: 2/2.
- Вода/маркетинг: 1/2 — фраза «деловой, краткий» ок, но «предполагаем KB актуальна» лучше заменить на проверку.
ТОП-правки: (1) указать порог confident; (2) добавить fallback при недоступности LLM; (3) описать off-ramp (как агент отключает AI).
```

### Шаг 6. Дальше по WF-3: C3 (симуляция), C4 (pre-mortem), C5 (Gherkin), эвал-набор — те же шаблоны.

**Мораль примера:** за один день — от интервью к утверждённому, self-reviewed PRO, готовому стать промптом и входом в прототип.

## 9. Чек-листы

### 9.1 Definition of Ready для PRO
- [ ] Проблема оцифрована (число/частота).
- [ ] Golden path = 3–5 шагов.
- [ ] AI-секции: tone, latency (P95), accuracy/grounding, safety.
- [ ] ≥3 edge cases с реакцией.
- [ ] 1 первичная + 1 anti-metric.
- [ ] Owner и next step на неделю.
- [ ] ≤500 слов.

### 9.2 Pre-mortem чек-лист (для AI-фичи)
- [ ] Что если LLM недоступен? (fallback)
- [ ] Что если confident высокий, но ответ неверный? (human approve + метрика)
- [ ] Prompt injection — где поверхность атаки? (guardrails)
- [ ] Excessive agency — может ли агент выполнить действие без подтверждения? (right-size permissions)
- [ ] Cost-of-agents vs rules — где правило дешевле и надёжнее?
- [ ] Privacy/retention — куда идут данные? (zero-retention для чувствительных)
- [ ] Eval-набор есть до разработки?

### 9.3 AI-feature launch checklist
- [ ] Эвалы (офлайн) зелёные; есть регресс-базлайн.
- [ ] Guardrails: input + output фильтрация.
- [ ] Мониторинг: latency P95, confident-распределение, wrong-answer-rate, cost/запрос.
- [ ] Human-in-the-loop для рискованных действий.
- [ ] Rollback (выключение фичи флагом) и kill-switch.
- [ ] Legal/privacy подписал (retention, PII).

## 10. Риски и антипаттерны

| Риск | Что это | Митигация |
|---|---|---|
| **Hallucination** | LLM выдумывает факты/цитаты/метрики | grounding (RAG), ссылки на источник, запрет «додумывать» в системном промпте, эвалы |
| **Automation bias** | люди верят AI-ответу без проверки | human approve для рискованных действий; показ confident; разметка «черновик, не истина» |
| **Prompt injection** (OWASP #1 для LLM-приложений) | злоумышленник через ввод меняет поведение модели | input-фильтрация, разделение доверенных/недоверенных данных, минимизация агентских прав |
| **Excessive agency** | агент делает действия, которые не должен | right-size permissions, confirmation для мутаций, allow-list действий |
| **Overreliance** | делегируют критические решения без fallback | всегда есть fallback на правило/человека; метрика качества |
| **Prompt/spec drift** | промпт, PRD, дизайн, тикеты разъезжаются | C6 регулярно; контракт = источник правды (OpenAPI) |
| **Cost-of-agents vs rules** | агент дорог и нестабилен там, где правило дёшево и точно | сначала правило; AI только там, где нужна вариативность/язык |

## 11. Где AI проигрывает классике

Критический раздел — чтобы не применять AI там, где он вреден.

- **Детерминированные правила точнее и дешевле.** Если поведение можно описать if/else и таблицей — не вставляй LLM. LLM оправдан там, где нужна вариативность естественного языка или нетривиальная классификация.
- **Финансы, биллинг, юридически значимые расчёты — RAG/правила обязательны, LLM только для рендеринга.** Числа должен считать код, не модель. AI здесь — интерфейс, не движок истины.
- **Доступы/безопасность/PII — не агенту.** Решение «может ли пользователь X видеть Y» принимает код с проверкой прав, а не LLM.
- **Discovery без контакта с людьми — пустота.** AI-синтез интервью не заменяет сами интервью; OST из галлюцинаций опаснее, чем отсутствие OST.
- **Спек без edge-cases — мина.** AI охотно генерирует golden path и так же охотно пропускает негативные сценарии. C4/C7 частично лечат, но не заменяют инженерное мышление.
- **Эвалы — обязательны, не опциональны.** AI-фича без тестового набора — это релиз вслепую; нельзя отличить улучшение от регресса.
